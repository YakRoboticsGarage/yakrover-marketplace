A Protocol for Verified Physical-World Task Execution

Problem

Digital service procurement is structured: API calls carry machine-readable specifications, delivery is verified algorithmically, and settlement occurs in milliseconds. Physical service procurement is unstructured: phone calls carry verbal scope, executor selection is relationship-based, delivery is verified by human inspection, and settlement follows a 30-60 day invoice cycle. No protocol exists that integrates machine-readable task specification, algorithmic executor matching, automated completion verification, and programmatic settlement for physical-world work.

Converging Conditions

Autonomous collection: DroneDeploy operates docked drones on 100+ projects (Oct 2025). Komatsu SMARTCONSTRUCTION: 35,000 sites in Japan (Nov 2025).

Digital design deliverables: Japan i-Construction: 86% ICT adoption in 6 years via mandate. Norway V770: 50%+ change order reduction with model-based delivery.

Numerical regulatory specs: MDOT Section 205.03.N: subgrade ±1 inch. Section 302.03.A: aggregate base ±½ inch. Published, deterministic, machine-parseable.

Workforce decline: Average US PLS age: 57-60 (ASCE/NSPS). 51% of engineering firms turning down work (ACEC Q4 2024). 8-year licensing pipeline. Equivalent shortages documented in Japan, Korea, Australia, Europe.

Agent tool-use: Model Context Protocol (Anthropic, 2024): thousands of tool servers. Agent-mediated procurement of physical services does not exist today. The infrastructure is being built.

Four Preconditions for Task Liquidity

A physical-world task becomes algorithmically procurable when four conditions are simultaneously met: structured specification (task describable in machine-readable terms), qualified executor discovery (credentials verifiable programmatically), machine-verifiable completion (algorithm determines success), and programmatic settlement (payment triggers on verified delivery). Construction survey on DOT highway projects satisfies all four with currently available technology.

Protocol Components and Development Status (v1.4.1 — Alpha, Functional)

The protocol is implemented as seven modules. The verification engine is the domain-specific component; all others are domain-agnostic and shared across verticals.

Coordination Layer: MCP-native interface (FastMCP), reverse auction engine (11-state lifecycle, 4-factor scoring, geographic filtering). Deployed on Fly.io. Alpha, functional.

Task Registry: 18 task categories with structured capability requirements (ASPRS accuracy, CRS EPSG, sensors, deliverables). RFP-to-spec conversion. Alpha, functional.

Executor Registry: ERC-8004 robot identity on Base mainnet + Sepolia. Discovery via The Graph. EAS attestation. 1 production + 100 demo registrations. Alpha, deployed on mainnet.

Credential Verification: SAM.gov Exclusions API (live). Bond verification against Treasury Circular 570 (live, 500+ sureties). PLS and insurance: schema-complete, stubs. SAM.gov and bond live.

Verification Engine: QA framework (4 levels, 8 category schemas). Geospatial surface comparison, specification database, and compliance reporting are next phase. QA framework functional; geospatial not implemented.

Settlement Layer: Stripe (authorize/capture), Connect Express, ACH, USDC on Base (EIP-3009, gasless). 4-mode abstraction designed. Mode 1 operational; modes 2-4 designed.

Robot Federation: MCPRobotAdapter, 9 category simulators on Fly.io. Dynamic tool resolution. 1 physical robot connected. Simulators operational; no production survey robots.

Ecosystem Integration

The protocol accepts measurement data from any source: DroneDeploy (docked drones, ground robots), Propeller Aero (AeroPoints, DirtMate), Trimble/SITECH (TBC exports, machine control), Komatsu/EARTHBRAIN (SMARTCONSTRUCTION), and Procore/Autodesk (workflow integration). Robot operators (FAA Part 107 drone operators, ground robot operators, survey firms) serve as the qualified executor pool. Vertical commercial products built on the protocol verify against regulatory requirements such as DOT specifications.

Roadmap

v1.4.1 (current): Protocol layer with simulated fleet. MCP interface, auction engine, on-chain identity, settlement, credentials. No verification engine.

v1.5 (next): Construction verification engine (geospatial ingestion, surface comparison, MDOT spec database, compliance reports). Settlement modes 2-4. Task encryption. First real operator.

v2.0: Multi-robot workflows (aerial + ground + GPR). Operator dashboard. Live credential APIs (PLS boards, insurance). Weather-aware scheduling. Agent-initiated payment.

v2.5-3.0: Mining/quarry specs. Infrastructure inspection (NBI). Privacy: ZK proofs (SP1), Horizen L3, BBS+ anonymous reputation.

Technical Challenges

Privacy: Task encryption via ECDH + AES-256-GCM. On-chain memos: H(request_id || salt) only. Settlement privacy: 4-mode abstraction, Horizen L3 under evaluation (EU AMLR Article 79 constraints). BBS+ reputation credentials planned.

Scale: Operator supply density uncertain in rural corridors. Credential APIs fragmented (50 state PLS boards). Trust bootstrapping requires managed curation (Brooks Act QBS culture). MDOT drone data acceptance not yet standard.
