Partner Brief — April 2026

A Protocol for Verified Physical-World Task Execution

Problem

Physical service procurement is unstructured and unavailable for agents. However, the technologies necessary to enable API-like purchase of physical actions are emerging.

Yak Robotics is building a protocol and application layer to allow for machine-readable task specification, algorithmic executor matching, automated completion verification, and programmatic settlement for physical-world work.

Four Preconditions for Task Liquidity

A physical-world task becomes algorithmically procurable when four conditions are simultaneously met: structured specification (task describable in machine-readable terms), qualified executor discovery (credentials verifiable programmatically), machine-verifiable completion (algorithm determines success), and programmatic settlement (payment triggers on verified delivery). Construction survey on Department of Transportation (DOT) highway projects satisfies all four with currently available technology.

Protocol Components and Development Status (v1.4.1 — Alpha, Functional)

The protocol is implemented as seven modules. The verification engine is the domain-specific component; all others are domain-agnostic and shared across verticals.

Coordination Layer: MCP-native interface, reverse auction engine (11-state lifecycle, 4-factor scoring, geographic filtering). Deployed on Fly.io. Alpha, functional.

Task Registry: 18 task categories with structured capability requirements (American Society for Photogrammetry and Remote Sensing accuracy classes, coordinate reference system codes, sensors, deliverables). Request for Proposal (RFP) to spec conversion. Alpha, functional.

Executor Registry: On-chain robot identity (ERC-8004 standard on Base). Discovery via The Graph indexing protocol. Ethereum Attestation Service (EAS) for platform attestation. 1 production + 100 demo registrations. Alpha, deployed on mainnet.

Credential Verification: SAM.gov Exclusions API (live). Bond verification against Treasury Circular 570 (live, 500+ sureties). PLS and insurance verification: schema-complete, stubs. SAM.gov and bond live.

Verification Engine: Quality assurance framework (4 levels, 8 category schemas). Geospatial surface comparison, specification database, and compliance reporting are next phase. QA framework functional; geospatial not implemented.

Settlement Layer: Stripe (authorize/capture), Connect Express, Automated Clearing House (ACH) bank transfer, USDC stablecoin on Base (gasless). 4-mode abstraction designed. Mode 1 operational; modes 2-4 designed.

Robot Federation: MCP-to-MCP robot adapter, 9 category simulators on Fly.io. Dynamic tool resolution. 1 physical robot connected. Simulators operational; no production survey robots.

Ecosystem Integration

The protocol accepts measurement data from any source: DroneDeploy (docked drones, ground robots), Propeller Aero (AeroPoints, DirtMate), Trimble/SITECH (TBC exports, machine control), Komatsu/EARTHBRAIN (SMARTCONSTRUCTION), and Procore/Autodesk (workflow integration). Robot operators (Federal Aviation Administration Part 107 drone operators, ground robot operators, survey firms) serve as the qualified executor pool. Vertical commercial products built on the protocol verify against regulatory requirements such as DOT specifications.

Roadmap

v1.4.1 (current): Protocol layer with simulated fleet. MCP interface, auction engine, on-chain identity, settlement, credentials. No verification engine.

v1.5 (next): Construction verification engine (geospatial ingestion, surface comparison, MDOT spec database, compliance reports). Settlement modes 2-4. Task encryption. First real operator.

v2.0: Multi-robot workflows (aerial + ground-penetrating radar + ground LiDAR). Operator dashboard. Live credential APIs (PLS boards, insurance). Weather-aware scheduling. Agent-initiated payment.

v2.5-3.0: Mining/quarry specs. Infrastructure inspection (NBI). Privacy: zero-knowledge proofs, anonymous reputation credentials.

Technical Challenges

Privacy: Task encryption via standard elliptic-curve key exchange + AES-256 encryption. On-chain memos use commitment hashes only (no plaintext task data). Settlement privacy: 4-mode abstraction, Horizen L3 under evaluation (EU Anti-Money Laundering Regulation Article 79 constraints). Anonymous reputation credentials planned.

Scale: Operator supply density uncertain in rural corridors. Credential APIs fragmented (50 state PLS boards). Trust bootstrapping requires managed curation (Brooks Act QBS culture). MDOT drone data acceptance not yet standard.
